<AccordionGroup>
  <Accordion title="What is Herald Protocol?">
    Herald Protocol is the infrastructure for the agentic economy on 0G Chain. It ships an x402 facilitator so any API, MCP server, or agent can accept onchain payments on 0G, an MPP SDK for direct server-side settlement without a facilitator, and a cross-chain payment router so agents holding funds on 0G can pay services on other chains without managing multiple wallets.
  </Accordion>

  <Accordion title="What is x402?">
    [x402](https://x402.org) is an open payment standard built around the HTTP `402 Payment Required` status code. A server returns payment requirements; a client signs a payment; a facilitator verifies and settles it onchain. See the [x402 docs](https://docs.x402.org) for the full protocol.
  </Accordion>

  <Accordion title="What is MPP?">
    [MPP](https://mpp.dev) (Machine Payments Protocol) is an open standard for machine-to-machine payments over HTTP. Like x402, it uses the HTTP `402 Payment Required` status code — but settlement happens directly between client and server, with no facilitator in the middle. The client either broadcasts an ERC-20 `transfer` (push mode) or signs an ERC-3009 `TransferWithAuthorization` (pull mode); the server verifies on-chain. Use MPP when you want direct server-side settlement. Use x402 when you want gasless onboarding and a hosted facilitator.
  </Accordion>

  <Accordion title="Do I need to run my own facilitator?">
    **For x402:** No. Point your server at `https://facilitator.heraldprotocol.xyz` and you're done. The facilitator is free to use and drop-in compatible with the x402 spec.

    **For MPP:** No facilitator is needed. The server verifies payments directly on-chain — either by checking transaction receipts (push mode) or by broadcasting `transferWithAuthorization` (pull mode).
  </Accordion>

  <Accordion title="Which networks are supported?">
    **x402 facilitator:** See [x402 Supported networks](/x402/supported-networks).

    **MPP:** See [MPP Supported networks](/mpp/supported-networks).
  </Accordion>

  <Accordion title="Which tokens can I accept?">
    **x402:** Bridged USDC (`USDC.e`) on 0G Mainnet via EIP-3009 `transferWithAuthorization`, plus any ERC-20 token via Permit2 gas sponsoring. Swap via [hub.0g.ai](https://hub.0g.ai/swap?network=mainnet). 0G testnet has no official stablecoin yet — a Herald Protocol EIP-3009-compatible test token is on the roadmap.

    **MPP:** Bridged USDC (`USDC.e`) on 0G Mainnet via ERC-3009 `TransferWithAuthorization` (pull mode), or any ERC-20 token via standard `transfer` (push mode).
  </Accordion>

  <Accordion title="Does the facilitator hold funds?">
    No. The x402 facilitator verifies signed payment payloads and submits them to 0G. It never custodies funds. Settlement goes directly from buyer to the seller's `payTo` address onchain.
  </Accordion>

  <Accordion title="Can I use any x402 or MPP SDK?">
    **x402:** Yes. The facilitator implements the standard `/verify`, `/settle`, and `/supported` endpoints, so any x402-compatible client or server library works. On the seller side, `@heraldprotocol/x402` wraps `@x402/evm` with 0G-aware money parsing so you can price endpoints in USD.

    **MPP:** Yes. `@heraldprotocol/mpp` provides native 0G support with direct server-side settlement — no facilitator required. The MPP protocol is framework-agnostic and works with any `mppx`-compatible adapter.
  </Accordion>

  <Accordion title="What is the Herald Router?">
    The Herald Router is a hosted proxy at `https://router.heraldprotocol.xyz` that lets agents holding funds on 0G pay any x402-protected service on another supported chain. The agent pays the router on 0G; the router pays the destination using its own wallet on the target chain. No bridging, no multi-chain wallet management. See [Router Overview](/router/overview).
  </Accordion>

  <Accordion title="Which chains does the router support?">
    Currently 0G Mainnet (`eip155:16661`) and Base Mainnet (`eip155:8453`). The agent always pays from 0G; the router settles on whichever chain the destination service requires.
  </Accordion>

  <Accordion title="Does the router hold funds?">
    The router holds pre-funded wallets on each supported destination chain to settle outbound payments. Your agent sends payment to the router on 0G; the router bridges that value by paying the destination on your behalf from its own wallet.
  </Accordion>

  <Accordion title="What's the difference between x402 and MPP?">
    Both solve machine payments over HTTP, but with different trade-offs:

    | | x402 | MPP |
    |---|---|---|
    | **Settlement** | Facilitator broadcasts | Direct client-server |
    | **Gas** | Facilitator pays | Client pays (push) or server pays (pull) |
    | **Token support** | EIP-3009 + Permit2 (any ERC-20) | Any ERC-20 (push) or ERC-3009 (pull) |
    | **Best for** | Gasless onboarding, hosted infra | Direct control, no third-party |

    Use x402 when you want a hosted facilitator and Permit2 support. Use MPP when you want direct settlement without a middleman.
  </Accordion>
